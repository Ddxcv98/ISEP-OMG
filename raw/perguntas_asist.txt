O DRP (“Disaster Recovery Plan”) é para o administrador de sistemas um dos elementos fundamentais do BCP (“Business Continuity Plan”)
A ativação do plano de contingência (“Contingency Plan”) é normalmente decidida pelos utilizadores
O valor do RPO (“Recovery Point Objective”) nunca pode ser inferior ao tempo que demora a realizara cópia de segurança (“backup”)
O RPO refere-se a um período de tempo decorrido após o desastre
A forma como as cópias de segurança são realizadas afeta diretamente o RPO, mas não tem qualquer efeito sobre o RTO
A “Politica de segurança” é um documento que contribui de forma significativa para a prevenção de falhas (“fault avoidance”)
Para garantir confidencialidade com uma cifra simétrica é necessário que o emissor e o recetor conheçam uma chave secreta que mais ninguém conhece
O número de bits usados das chaves criptográficas secretas e privadas não afeta de forma significativa a segurança que se obtém
A criptografia assimétrica (chave pública) nunca pode garantir a autenticidade, para isso é necessário recorrer sempre a criptografia simétrica
Quando uma mensagem é cifrada com a chave privada de uma cifra assimétrica, a confidencialidade não fica garantida
Quando se aplica uma função “hash“ a uma mensagem, o resultado é um código cujo comprimento é sempre igual ao comprimento da mensagem
O protocolo PAP (“Password Authentication Protocol”) só deve ser usado em canais que garantam a confidencialidade das transações
Uma vantagem do CHAP (“Challenge Handshake Authentication Protocol”) relativamente ao PAP é que garante a autenticidade do serviço
Dos vários componente da arquitetura AAA, o PEP(“Policy Enforcement Point”) é o único com o qual os utilizadores interagem diretamente
O PIP (“Policy Information Point”) é o responsável por registar os acessos ao sistema (“accounting”)
O PDP (“Policy Decision Point”) pode consideraras credenciais de autenticação válidas, mas mesmo assim negar o acesso ao serviço
O protocolo LDAP (“Lightweight Directory AccessProtocol”) pode ser usado nas comunicações entre o PDP e o PIP
É possível criar um objeto LDAP que pertença a várias classes auxiliares, sem pertencer a nenhuma classe estrutural
Em LDAP a definição de classes suporta herança, por isso é possível definir uma nova classe tendo como base uma classe já existente
Numa base de dados LDAP é possível existirem vários objetos com o mesmo RDN (“Relative Distinguished Name”)
No sistema KERBEROS os TGT (“Ticket Granting Ticket”) são fornecidos aos “principal”, mas não existe nenhum “principal” capaz de os decifrar
As cifras assimétricas (chave pública) são a base de funcionamento do sistema KERBEROS, sem elas este sistema não pode operar
Quando um nome de um “realm” KERBEROS deriva de um nome de domínio DNS deve ser expresso em maiúsculas
O AES (“Advanced Encryption Standard”) é uma das cifras suportadas pelo KERBEROS
A mensagem “ClientHello” enviada pelos clientes TLS (“Transport Layer Security”) indica qual é a versão mais recente de TLS que o cliente suporta
O “cipher suite” “TLS_RSA_WITH_AES_256_CBC_SHA256” realiza a autenticação através de certificados de chave pública
O acesso da “intranet” à DMZ (“DeMilitarized Zone”) deve ser filtrado permitindo a passagem apenas do tráfego estritamente necessário
Para combater ataques de “spoofing” de endereços IPv4 os “firewall” estáticos são suficientes
O processo de execução de ataques do tipo MITM (“Man-in-the-middle”) inclui normalmente ataques de“sniffing”
Os ataques designados “SYN flood” obrigam o atacante a “adivinhar” os números de sequência usados pelo servidor
Se for usado exclusivamente ARP estático podemos evitar os ataques de “spoofing” ao protocolo ARP (“Address Resolution Protocol”)
Seja qual for o tipo de serviço TCP considerado pode sempre sofrer um ataque do tipo DoS (“Denial of Service”) designado “SYN flood”
Os ataques DDoS refletidos apenas são possíveis se o atacante tiver a possibilidade de realizar “spoofing” IP
O IPsec (“Internet Protocol Security”) proporciona segurança ao protocolo IPv6, mas não é compatível com o IPv4
O “Encapsulating Security Payload” (ESP) nunca pode ser usado isoladamente, só funciona em conjunto com o AH (“Authentication Header”)
Seja em “Tunnel Mode” ou “Transport Mode”, o AH (“Authentication Header”) não garante a confidencialidade, os dados ficam visíveis para os intrusos
Todas as implementações IPsec devem suportar o ESP, o suporte de AH não é obrigatório, mas o AH e o ESP podem ser usados em conjunto.
No IPsec, para se conseguir uma comunicação segura nos dois sentidos entre duas entidades, são necessárias duas SA (“Security Association”)
O algoritmo “Diffie-Hellman” é usado pelo IKE (“Internet Key Exchange”) para criar a primeira SA, designada IKE SA
As VPN do tipo LAN-LAN são criadas pelos administradores das redes, os utilizadores não têm consciência de que estão a ser utilizadas
O “Layer 2 Tunneling Protocol” (L2TP) pode ser usado em conjunto com o IPsec para criar uma VPN do tipo LAN-LAN
Os pacotes RTP (“Real Time Protocol”) transportam uma etiqueta data/hora que permite o sincronismo dos fluxos
A compressão de cabeçalhos (“header compression”) deve ser aplicada quando a quantidade de dados em cada pacote é reduzida
A técnica LFI (“Link Fragmentation and Interleaving”) é aplicada apenas aos pacotes de grande dimensão
No protocolo TCP, quando o valor do RTT (“Round-Trip Time”) aumenta, o valor do RTO (“Retransmission TimeOut”) é automaticamente diminuído
No mecanismo RED (“Random Early Detection”) enquanto a fila de saída estiver vazia, nenhum pacote é descartado
Os bits de precedência IP são usados para marcar os pacotes, podem assumir qualquer valor de zero a 4096
A técnica de gestão de filas “Priority Queuing” (PQ) tem a vantagem de dar totais garantias para o tráfego considerado mais importante
Na gestão de filas “Custom Queuing” (CQ), algum tráfego não prioritário é processado mesmo que exista tráfego prioritário nas filas
O CAR (“Committed Access Rate”) não é uma técnica do tipo SOFT QoS (“Quality of Service”)
No CAR dos “routers” CISCO, ao tráfego que está dentro do “Normal Burst” não é aplicada qualquer ação
O protocolo RADIUS é usado nas comunicações entre o PEP e o PIP (“Policy Information Point”)
